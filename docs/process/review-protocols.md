# Review Protocols — Felix & Alex Story Universe

*How to run reviews consistently, and — critically — how the Orchestrator adjudicates conflicting notes so you FIX the right things instead of applying every note (which produces mush). Pairs with `personas.md`.*

---

## The three review types

### 1. Standard Panel Pass
Run **Panel A** (five personas). Each gives their SHARPEST note, not a list. Look for **convergence** — where ≥2 personas independently flag the same thing, that's the real issue. Then orchestrate.

**When:** every new or rewritten story draft; every narration script before ElevenLabs.

### 2. Adversarial Review
Run **Panel B** (four hostile reviewers) AFTER a Standard Pass has been broadly passed. Their job is to break it. Expect over-correction pressure — resist it.

**When:** before locking anything expensive (narration), and on any bible/frame before committing to a multi-item build.

### 3. Pre-Mortem
Run persona 11 alone, or as part of Panel C. "It's N months later; this failed; why?" List plausible causes, sort fixable-now vs. accept-as-risk.

**When:** foundation documents (bibles, frames, slates), and any time a big commitment is about to be made.

## THE ORCHESTRATOR (the important part)

After any review, ONE adjudication step decides what actually changes. Never skip this. Never apply all notes.

### Step 1 — Collect every note, tagged by persona and weight.

### Step 2 — Sort each note into exactly one bucket:

**FIX** — apply it. Criteria (any one):
- ≥2 HIGH-weight personas independently flagged the same thing.
- It's a **safety** note (Skeptical Parent, or Historian on a real-world fact) that's genuinely true.
- It's a **Continuity Auditor** bug that is demonstrably a real error.
- It's a cheap change that clearly improves the piece with no cost to a deliberate design choice.

**MITIGATE** — partially address, lightly. Criteria:
- A real concern, but a full fix would over-correct or harm a deliberate choice.
- Example: "too gentle" → add friction to *some* books, don't make it an action series.

**HOLD (accepted trade-off)** — do NOT change; record why. Criteria:
- The note asks for a fundamentally different artifact than the one deliberately chosen (most Cynical-Editor "add a villain" notes).
- It's a delivery/production note, not a text problem (→ route to production notes instead).
- It's an inherent, accepted property (e.g. a finale is calmer than the opener).

**ACTIVELY DECIDE** — escalate to the human. Criteria:
- A genuine fork with no clearly-right answer (scope, structure, how far to push an edge).
- Highest-probability pre-mortem failures that aren't text fixes (e.g. "built then not finished" = a scope decision).

### Step 3 — State the adjudication explicitly.
For each note: bucket + one-line reason. This is the record of *why* something was or wasn't changed — it prevents re-litigating and prevents silent over-correction.

### Step 4 — Apply only the FIX and MITIGATE items. Then note HOLDs and escalate DECIDEs.

### Step 5 — Re-review only if a FIX was structural. Polish-level fixes don't need a fresh panel; structural ones do.

## Orchestrator principles (learned the hard way)

1. **Convergence beats volume.** One persona repeating a point five ways is one note. Three personas hitting it once is the signal.
2. **Resist adversarial over-correction.** The Cynical Editor is usually right that it's *safe* and wrong about the *fix*. Extract the real weakness; reject the "make it a different story" prescription.
3. **Safety and continuity are near-non-negotiable.** These get fixed even at weight cost.
4. **Protect deliberate design choices by name.** If the piece is deliberately the calm/grounded one, "too gentle" is partly the point — mitigate, don't capitulate.
5. **The characteristic failure mode of this universe is "narrator says the quiet part out loud."** It recurs. Watch for it every single pass; cutting it is almost always right.
6. **Distinguish text problems from production/parenting problems.** "The real world must cash the check" and "narrate the seams slower" are not edits — route them, don't jam them into prose.
7. **Cheap-now, expensive-later bias.** Anything trivial to fix pre-narration that would be costly post-narration gets fixed now.

## Standard sequence for taking a story from draft to locked

1. Draft against the bible (check ledgers first for current state).
2. **Standard Panel Pass** (A) → Orchestrate → apply fixes.
3. Read/react (human) — the ear catches what analysis misses.
4. **Adversarial Review** (B) → Orchestrate → apply fixes.
5. **Whole-work / cold-listen check** if part of a series (does it land for someone who only heard this one?).
6. Lock prose.
7. Produce narration script (strip to v2, retain punctuation-pacing — see `narration-production-notes.md`).
8. **Audio Ear pass** on the script.
9. Generate audio; update ledgers.
